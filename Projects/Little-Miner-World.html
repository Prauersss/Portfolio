About the Game

A 3D mining RPG made in Unity: dig through a procedural voxel world, collect minerals, upgrade your pickaxe and chat with the locals along the way.

The Idea

Little Miner World is a voxel mining RPG where the terrain is fully destructible — it's generated with the Marching Cubes algorithm, so you can dig wherever you want. The loop is simple: mine minerals, fill out your codex, do missions for the NPCs, and save up for pickaxe upgrades so you can dig deeper.

Everything runs on systems I wrote in Unity with C#, from the voxel mesh generation down to the UI for the codex and the shop, the dialogue, the cinematics and the save system.

Main Systems

Marching Cubes Terrain

My own Marching Cubes implementation generates the destructible voxel terrain, and the mesh rebuilds in real time as you mine through it.

Pickaxe Mining

The pickaxe tip detects hits against the terrain physically — each strike spawns particles and drops pickups depending on which mineral you hit.

Inventory & Pickups

Items drop as 3D objects in the world and go into a single inventory with stackable materials, which the shop and missions read from directly.

Mineral Codex

An in-game encyclopedia that unlocks entries as you discover new minerals, with an unlock banner and a detail page for each one.

Shop & Upgrades

You spend the materials you've mined to upgrade your pickaxe. Each shop item shows its stats and requirements, with a little card animation on purchase.

NPC Dialogue

Conversations are defined in ScriptableObjects, including their effects and end triggers, and the text animates on screen using Text Animator.

Missions

A mission panel keeps track of your current objectives. Missions are data-driven through ScriptableObjects and update as you make progress.

Cinematics & Saving

Key story moments trigger short cinematics, and a save system keeps your progress, inventory and discovered minerals between sessions.

Problems I Ran Into

Real-Time Voxel Mesh Updates

Procedural Generation
The problem

At first I was regenerating the whole Marching Cubes mesh on every pickaxe hit. That caused visible frame hitches, and mining felt laggy as a result.

How I fixed it

I split the terrain into chunks so only the chunk you hit gets remeshed, with dirty flags marking which ones actually need an update. Nothing else gets recalculated.

Accurate Pickaxe Hit Detection

Physics & Interaction
The problem

Normal collision detection against a mesh that keeps changing gave me both phantom hits and missed swings, depending on the angle and how fast the player was moving.

How I fixed it

I gave the pickaxe tip its own component that does a small sphere overlap each frame, but only during the swing animation window, and picks the closest voxel it finds. Much more reliable.

Codex Unlock State Persistence

Data & Save System
The problem

The codex had to remember which entries you'd unlocked between play sessions, but I couldn't tie that data to scene objects that get destroyed on scene changes.

How I fixed it

The entries live in ScriptableObjects and their locked/unlocked state gets saved to player prefs, then loaded and applied to the UI when the game starts.

Dynamic Shop Inventory

UI & Inventory Systems
The problem

The shop had the same issue: its inventory depends on player progress, and keeping it in scene objects meant losing data every time a scene reloaded.

How I fixed it

I made the shop data-driven too: the inventory is defined in ScriptableObjects and the UI reads from them directly, so it survives scene reloads and updates on its own.
